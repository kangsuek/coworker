 메시지 전송 후 처리 흐름                                                                                                                                                                                  
                                                                  
  1. 사용자 입력 (UserChannel/index.tsx)

  Enter 키 또는 전송 버튼 클릭
    ↓
  handleSend()
    ├─ input 공백 확인, 실행 중 여부 확인 (submitting || runId !== null)
    ├─ UI에 user 메시지 즉시 표시 (낙관적 업데이트)
    └─ api.chat({ session_id, message, llm_provider, llm_model }) 호출
    ※ 실행 중에는 '취소 ✕' 버튼이 활성화됨 (handleCancel 호출 시 /api/runs/{run_id}/cancel 호출)

  2. API 요청 (chat.py)

  POST /api/chat { session_id, message, llm_provider, llm_model }
    ↓
  session_id가 있고 provider/model이 다르면 → 세션 설정 업데이트
  session_id가 없으면 → 새 세션 자동 생성 (기본값: claude-cli)
    ↓
  create_user_message(session_id, "user", message)
    ├─ user_messages 테이블에 저장
    └─ session.title이 없으면 첫 30자로 자동 설정
    ↓
  create_run(session_id, user_message_id)
    └─ runs 테이블에 status="queued", started_at=NOW 로 생성
    ↓
  background_tasks.add_task(_run_reader_agent, ...)  ← 백그라운드로 즉시 반환
    ↓
  { run_id, session_id } 즉시 반환 (200ms 이내)

  3. 프론트엔드 폴링 시작 (useRunPolling.ts)

  onRunChange(run_id) → currentRunId 설정
    ↓
  useRunPolling(runId) 활성화
    ↓
  2초 ± 300ms 간격으로 GET /api/runs/{run_id} 폴링
    └─ status 변화 감지: queued → thinking → solo|delegating → done|error|cancelled
    └─ timing 정보 수신: thinking_started_at, cli_started_at, finished_at 등

  4. 백그라운드: ReaderAgent 처리 (reader.py)

  _run_reader_agent(session_id, user_message, run_id)
    ↓
  update_run_status("thinking", thinking_started_at=NOW)
    ↓
  _classify(user_message)
    └─ 규칙 기반 판정 (classify_message.py): TEAM_TRIGGER_HEADER 확인 후 즉시 판정 (CLI 호출 제거됨)
    ↓
  모드 결정 (Solo 또는 Team)

  4-A. Solo 모드

  update_run_status("solo", cli_started_at=NOW)
    ↓
  _solo_respond(user_message)
    └─ 선택된 LLM Provider를 통해 스트리밍 생성
    ↓
  db.get(Run, run_id) ← 세션 삭제 여부 확인 (안전장치)
    ↓
  create_user_message(session_id, "reader", text, mode="solo")
  update_run_status("done", response=text, mode="solo", finished_at=NOW)

  4-B. Team 모드

  update_run_status("delegating", mode="team")
    ↓
  [Agent 순차 실행 (1~5개)]
    ↓
    각 Agent마다:
    ├─ update_run_status("working", progress="1/3")
    ├─ _assemble_context() ← 이전 결과 3000자 초과 시 요약 CLI 호출
    ├─ create_agent_message() ← Agent Channel DB 레코드 생성
    ├─ LineBufferFlusher 시작 (0.5초 배치)
    │    └─ 동기 sqlite3 연결(WAL 모드)을 사용하여 실시간 DB 쓰기 (aiosqlite 제한 회피)
    ├─ agent.execute() ← 선택된 LLM Provider 호출
    │    └─ on_line 콜백으로 실시간 stdout → flusher.append()
    ├─ flusher.stop() ← 잔여 버퍼 최종 flush
    ├─ update_agent_message_content() ← 최종 content 동기화 (aiosqlite)
    └─ update_agent_message_status("done")
    ↓
  update_run_status("integrating")
    ↓
  _integrate_results() ← 모든 Agent 결과 통합 호출
    ↓
  db.get(Run, run_id) ← 세션 삭제 여부 확인 (안전장치)
    ↓
  create_user_message(session_id, "reader", final, mode="team")
  update_run_status("done", response=final, mode="team", finished_at=NOW)

  5. 폴링 상태 변화 감지 및 UI 갱신

  useRunPolling → 상태 변화에 따라 콜백 호출
    ↓
  onDone(response, mode, model, timing)
    ├─ addMessage({ role: "reader", content: response, timing, ... }) → 채팅창에 답변 표시
    ├─ onModeChange(mode) → currentMode 업데이트
    └─ onRunChange(null) → runId 초기화, 폴링 종료
    ↓
  onCancelled()
    ├─ Team 모드인 경우 "N개 중 M개 Agent 완료 후 취소됨" 메시지 표시
    └─ onRunChange(null) → runId 초기화, 폴링 종료

  6. Team 모드 전용: Agent Channel 폴링 (useAgentPolling.ts)

  currentMode="team" + currentRunId 있으면 활성화
    ↓
  2초 ± 300ms 간격으로 GET /api/runs/{run_id}/agent-messages
    └─ Agent 메시지 목록을 덮어쓰기 방식으로 갱신
    ↓
  AgentChannel 컴포넌트에서 각 Agent의 실시간 작업 내용 표시 (LineBufferFlusher에 의해 실시간 갱신됨)

  전체 타임라인 요약

  [0ms]    사용자 Enter
  [~50ms]  POST /api/chat → run_id 반환, 백그라운드 시작
  [~60ms]  백그라운드: status=thinking (규칙 기반 분류, 즉시 완료)
  [~100ms] useRunPolling 폴링 시작 (2초 간격)
  [~2s]    첫 폴링: status=solo 또는 working (이미 실제 작업 중)
  [완료 시] status=done → UI에 답변 및 실행 시간(timing) 표시, 폴링 종료
