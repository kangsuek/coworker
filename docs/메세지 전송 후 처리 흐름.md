 메시지 전송 후 처리 흐름                                                                                                                                                                                  
                                                                  
  1. 사용자 입력 (UserChannel/index.tsx:97-129)                                                                                                                                                             
                                                                                                                                                                                                            
  Enter 키 또는 전송 버튼 클릭
    ↓
  handleSend()
    ├─ input 공백 확인, 실행 중 여부 확인 (submitting || runId !== null)
    ├─ UI에 user 메시지 즉시 표시 (낙관적 업데이트)
    └─ api.chat({ session_id, message }) 호출

  2. API 요청 (chat.py:35-51)

  POST /api/chat { session_id, message }
    ↓
  session_id 없으면 → 새 세션 자동 생성
    ↓
  create_user_message(session_id, "user", message)
    ├─ user_messages 테이블에 저장
    └─ session.title이 없으면 첫 30자로 자동 설정
    ↓
  create_run(session_id, user_message_id)
    └─ runs 테이블에 status="queued"로 생성
    ↓
  background_tasks.add_task(_run_reader_agent, ...)  ← 백그라운드로 즉시 반환
    ↓
  { run_id, session_id } 즉시 반환 (200ms 이내)

  3. 프론트엔드 폴링 시작 (useRunPolling.ts)

  onRunChange(run_id) → currentRunId 설정
    ↓
  useRunPolling(runId) 활성화
    ↓
  2초 ± 300ms 간격으로 GET /api/runs/{run_id} 폴링
    └─ status 변화 감지: queued → thinking → solo|delegating → done|error|cancelled

  4. 백그라운드: ReaderAgent 처리 (reader.py)

  _run_reader_agent(session_id, user_message, run_id)
    ↓
  update_run_status("thinking")
    ↓
  _classify(user_message)
    ├─ 1차: 규칙 기반 판정 (명확한 team이면 CLI 호출 생략)
    └─ 2차: haiku CLI 호출로 Solo/Team 판정
         └─ execute_with_lock() → Global Lock 획득 후 실행

  4-A. Solo 모드

  update_run_status("solo")
    ↓
  _solo_respond(user_message)
    └─ Claude CLI 호출 (execute_with_lock)
    ↓
  db.get(Run, run_id) ← 세션 삭제 여부 확인 (안전장치)
    ↓
  create_user_message(session_id, "reader", text, mode="solo")
  update_run_status("done", response=text, mode="solo")

  4-B. Team 모드

  update_run_status("delegating", mode="team")
    ↓
  [Agent 순차 실행 (1~5개)]
    ↓
    각 Agent마다:
    ├─ update_run_status("working", progress="1/3")
    ├─ _assemble_context() ← 이전 결과 3000자 초과 시 요약 CLI 호출
    ├─ create_agent_message() ← Agent Channel DB 레코드 생성
    ├─ LineBufferFlusher 시작 (0.5초 배치 DB 쓰기)
    ├─ agent.execute() ← Claude CLI 호출 (execute_with_lock)
    │    └─ on_line 콜백으로 실시간 stdout → flusher.append()
    ├─ flusher.stop() ← 잔여 버퍼 최종 flush
    ├─ update_agent_message_content() ← 최종 content 동기화
    └─ update_agent_message_status("done")
    ↓
  update_run_status("integrating")
    ↓
  _integrate_results() ← 모든 Agent 결과 통합 CLI 호출
    ↓
  db.get(Run, run_id) ← 세션 삭제 여부 확인 (안전장치)
    ↓
  create_user_message(session_id, "reader", final, mode="team")
  update_run_status("done", response=final, mode="team")

  5. 폴링이 done 감지 → UI 완성 (UserChannel/index.tsx:46-56)

  useRunPolling → status="done" 감지
    ↓
  onDone(response, mode, model) 콜백
    ├─ addMessage({ role: "reader", content: response }) → 채팅창에 답변 표시
    ├─ onModeChange(mode) → currentMode 업데이트
    └─ onRunChange(null) → runId 초기화, 폴링 종료
    ↓
  useSession.addMessage()
    └─ role="reader"이면 300ms 디바운스 후 세션 목록 갱신 (제목 업데이트 반영)

  6. Team 모드 전용: Agent Channel 폴링 (useAgentPolling.ts)

  currentMode="team" + currentRunId 있으면 활성화
    ↓
  2초 ± 300ms 간격으로 GET /api/runs/{run_id}/agent-messages
    └─ Agent 메시지 목록을 덮어쓰기 방식으로 갱신
    ↓
  AgentChannel 컴포넌트에서 각 Agent의 실시간 작업 내용 표시
    └─ status: working(진행중) / done(완료) / error(오류)

  전체 타임라인 요약

  [0ms]    사용자 Enter
  [~50ms]  POST /api/chat → run_id 반환, 백그라운드 시작
  [~50ms]  useRunPolling 폴링 시작 (2초 간격)
  [~2s]    첫 폴링: status=thinking (분류 중)
  [~4s]    Solo: status=solo → 응답 생성 중
           Team: status=delegating → Agent 순차 실행
  [완료 시] status=done → UI에 답변 표시, 폴링 종료
